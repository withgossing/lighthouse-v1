# 요구사항 정의서 (Requirements Specification v1.1)

본 문서는 사내 IT Helpdesk 시스템(Lighthouse) 구축을 위한 기능 요구사항을 정의합니다. 특히 **업무 분류(Category)**, **티켓 전달 및 이관(Routing/Transfer)**, **전체 처리 흐름 조회(History Log)** 에 초점을 맞추어 작성되었습니다.

---

## 1. 업무 분류 (Categories) 체계화 및 열람 권한 차등
단순 나열식 카테고리를 탈피하고, 2단계 깊이(대분류 -> 중분류)를 가진 트리 형태로 개편하며 관리자가 직접 관리할 수 있어야 합니다.

### 1-1. 2단계 카테고리(Category) 구조
*   **시스템 문의 (System IT)**: 인프라 및 기반 시스템 이슈
    *   `네트워크` (VPN 접속 불가, 인터넷 끊김 등)
    *   `어플리케이션` (PC 소프트웨어 오작동, 라이센스 만료 등)
    *   `하드웨어/장비` (PC 고장, 모니터 교체 등)
    *   `계정/권한` (사내 시스템 계정 생성 및 권한 부여 요청)
*   **독립/업무 시스템 문의 (Business Applications)**: 회사 비즈니스 로직과 직접 연결된 시스템 이슈
    *   `금융상품 관리 시스템` (상품 등록/변경/조회 오류 등)
    *   `매매 시스템` (주문 지연, 체결 내역 불일치 등)
    *   `계좌 관리 시스템` (계좌 개설/해지/조회 오류 등)
    *   `출납 관리 시스템` (입출금 지연, 정산 오류 등)

### 1-2. 사용자 권한에 따른 카테고리 노출 차등 부여
*   **개념**: 모든 사용자가 모든 문의 카테고리를 볼 수 있는 것이 아니라, 사용자의 부서(Department)나 직급(Role)에 따라 화면에 노출되는 업무 분류를 제한합니다.
*   **요구사항**:
    *   예: 일반 영업 사원은 `시스템 문의`와 `매매 시스템` 카테고리만 선택 가능. 재무팀 직원은 `출납 관리 시스템` 카테고리를 추가로 선택 가능.
    *   시스템 관리자는 관리자 화면에서 "어떤 카테고리가 어떤 권한/부서에게 열람(선택) 가능한지" 매핑 설정을 할 수 있어야 합니다.

## 2. 티켓 전달(Transfer) 및 담당자 지정 기능

본인의 담당 업무가 아니라고 판단되었을 때, 티켓을 타인에게 "전달(패스)"하는 기능입니다. (기존 전화 돌리기의 시스템화)

### 2-1. 담당자 이관 (Transfer Ticket)
*   **요구사항**: 
    1.  현재 티켓의 담당자(Admin)는 티켓 상세화면에서 **[담당자 변경/전달]** 버튼을 통해 자신에게 할당된 티켓을 다른 관리자(혹은 다른 부서 그룹)에게 넘길 수 있습니다.
    2.  전달할 때 필수적으로 **"전달 사유(Transfer Note)"** 를 입력해야 합니다. (예: "이 건은 인프라팀 소관이 아니라 금융상품팀 소관으로 보입니다.")
    3.  이관된 내역은 티켓의 타임라인(히스토리)에 명확히 기록되어야 합니다.

### 2-2. 수동 및 자동 배정 (Assignment)
*   **미할당 큐(Unassigned Queue)**: 새로 등록된 티켓은 기본적으로 담당자가 지정되지 않은 공용 큐에 담깁니다.
*   **수동 배정(Manual Assignment)**: 팀장 역할(Admin)이 미할당 큐를 보며 직접 담당자를 지정(Assign)하거나, 관리자가 스스로 [본인 앞으로 가져오기 (Take it)] 버튼을 눌러 담당자가 될 수 있습니다.
*   **자동 배정(Auto Routing - 고도화 범위)**: 특정 카테고리(예: '네트워크')의 티켓이 접수되면, 사전에 설정된 네트워크 팀의 관리자 중 현재 처리중인 티켓 수가 가장 적은 사람에게 자동 할당(Round-robin with workload check)됩니다.

## 3. 티켓 전체 처리 흐름(History & Timeline) 조회

티켓이 생성된 순간부터 해결(Closed)될 때까지의 **모든 상태 변경 내역(Audit Trail)**을 타임라인 형태로 시각화합니다. (투명성 확보 및 책임 소재 명확화)

### 3-1. 타임라인(Timeline) 로깅 대상 이벤트
*   **생성**: `홍길동님이 티켓을 접수했습니다. (2025-10-24 10:00)`
*   **담당자 지정**: `미할당 상태에서 [김철수 관리자]가 담당자로 지정되었습니다. (2025-10-24 10:30)`
*   **우선순위/상태 변경**: `김철수 관리자가 상태를 [접수(Open)]에서 [진행 중(In Progress)]으로 변경했습니다. (2025-10-24 10:35)`
*   **담당자 이관(Transfer)**: `김철수 관리자가 [이영희 관리자]에게 티켓을 이관했습니다. 사유: "매매 시스템 백엔드 로직 확인 필요" (2025-10-24 11:00)`
*   **고객 소통(Comment/Reply)**: (관리자/사용자 간의 메시지 왕복 내용 기록)
*   **추가 파일 첨부**: `이영희 관리자가 [분석결과_로그.txt] 파일을 첨부했습니다.`
*   **해결 처리**: `이영희 관리자가 티켓을 [해결됨(Resolved)] 으로 처리했습니다. (2025-10-24 13:00)`

### 3-2. 요구사항 (UI/UX)
*   일반 직원(User)과 관리자(Admin) 모두 티켓 상세 화면에 들어가면 좌측이나 하단에 이 히스토리가 **시간 역순(최신순) 혹은 정순**으로 일목요연하게 표시되어야 합니다.
*   이를 통해 직원은 내 문의가 어떻게 처리되고 있는지 투명하게 알고, 관리자는 이전에 누가 무슨 작업을 했는지 즉각 파악할 수 있어야 합니다.

## 🚀 다음 단계 (Action Items)
이 요구사항을 코드로 구현하기 위해서는 다음의 DB 스키마 수정이 필요합니다.

1.  기존 `enum TicketCategory` 를 삭제하고, **계층형 `Category` 테이블** (id, name, parentId) 신설.
2.  사용자 권한/부서와 Category 간의 **N:M 열람 권한 매핑 테이블(`CategoryAccess`)** 신설.
3.  단순 Comment 테이블 외에, 시스템 상태 변경을 로깅하는 전용 **`TicketHistory` (Audit Log) 테이블** 신설.

위 내용으로 이어서 DB 스키마 업데이트 작업을 즉각 진행할 수 있습니다.
