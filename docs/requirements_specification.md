# 요구사항 정의서 (Requirements Specification)

본 문서는 현재 구현되어 있는 사내 IT Helpdesk 시스템(Lighthouse)의 아키텍처 및 기능 명세를 바탕으로, **5개 전문가 에이전트(PM, 디자이너, 프론트엔드, 백엔드, QA/보안)의 검토 의견을 종합하여 도출한 차기 핵심 요구사항(Next-step Requirements)**을 정의합니다.

이번 요구사항은 단순히 기능을 정의하는 것을 넘어, 현재 구축된 MVP 시스템이 엔터프라이즈 환경에서 안정적으로 스케일업(Scale-up)하기 위해 반드시 해결해야 할 실무적, 기술적 과제들을 포함합니다.

---

## 1. 업무 분배 자동화 (Auto-Routing & Workload Management)

현재 관리자가 미할당(Unassigned) 큐에서 티켓을 수동으로 본인에게 가져가거나 배정하는 방식은 운영 효율성을 저하시킬 수 있습니다.

### 1-1. 스킬 기반(Skill-based) 라운드로빈 자동 할당 (PM 에이전트 제언)
*   **요구사항**: 신규 티켓 생성 시 카테고리(Category)와 각 관리자의 주특기(Skillset)를 매핑하여, 가장 적합한 관리자에게 자동으로 티켓을 1차 배정해야 합니다.
*   **추가 조건**: 만약 특정 관리자의 `상태(Status)`가 자리 비움이거나 현재 `진행 중(IN_PROGRESS)` 인 티켓 개수가 임계값(Threshold)을 넘는 경우, 차순위(혹은 부하가 적은) 관리자에게 우회(Fallback)하여 배분하는 로직이 필요합니다.

---

## 2. 보안 및 파일 업로드 통제 (File Attachment Security)

QA 및 보안 에이전트의 관점에서, 현재 지원되는 파일 첨부 및 공유 기능은 사이버 보안 취약점의 통로가 될 수 있습니다.

### 2-1. 파일 처리 보안 정책 강화 (QA/보안 에이전트 제언)
*   **요구사항**: 사용자가 에러 로그나 캡처 화면을 `[첨부 파일]` 로 업로드할 때, 철저한 확장자 및 해시 검증이 선행되어야 합니다.
*   **구현 스펙**:
    *   **White-list 방식 도입**: `.exe`, `.sh`, `.bat` 등 실행 가능한 악성 스크립트 업로드를 원천 차단하고, `.png`, `.jpg`, `.pdf`, `.txt` 등 허용된 포맷(Mime-Type)만 백엔드(NestJS Controller/Pipe 레벨)에서 승인해야 합니다.
    *   **용량 제한**: 첨부 파일 하나당 Max Size(예: 10MB) 제한 정책을 마련하여 스토리지 어뷰징을 방지해야 합니다.

---

## 3. UI/UX 및 반응형 경험 강화 (Responsive Layouts)

다양한 디바이스와 환경에서 시스템에 접근하는 사용자 풀을 고려할 때, 현재 데이터가 집약된 화면(Data Table)에 대한 반응형 다변화가 필요합니다.

### 3-1. 모바일 뷰 최적화 (UI/UX 에이전트 제언)
*   **요구사항**: 관리자 및 일반 사용자가 스마트폰으로 이슈를 접수하거나 처리 현황을 볼 때 화면 레이아웃이 깨지지 않아야 합니다.
*   **구현 스펙**:
    *   데이터 테이블(`DataTable`)의 컬럼 중 `ID`, `Category`, `Assignee` 같은 부차적 정보들을 모바일 뷰(`md` 브레이크포인트 미만)에서는 숨김 처리(`hidden md:table-cell`).
    *   대신, `Title`과 `Status(뱃지)` 위주의 압축된 카드(Card) 뷰 형태로 제공하여 모바일 터치(Touch) 및 스크롤 경험을 개선해야 합니다.

---

## 4. 실시간성 확보 및 프론트엔드 최적화 (Real-time Communications)

빠른 IT 문제 해결을 위해 커뮤니케이션(Ticket Comments) 속도를 실제 사내 메신저 수준으로 끌어올릴 필요가 있습니다.

### 4-1. 실시간 대화 지원 (프론트엔드 에이전트 제언)
*   **요구사항**: 티켓의 상태(Status/Priority)가 변경되거나, 누군가 새로운 댓글(Comment)을 작성했을 때 페이지를 새로고침(F5) 하지 않고도 해당 내용이 즉시 화면에 반영(Optimistic UI)되어야 합니다.
*   **도입 방안**: 현재 REST API 구조 위에 `WebSocket` (혹은 Next.js Server-Sent Events)을 추가로 도입하여 해당 티켓을 보고 있는 유저들에게 실시간 브로드캐스팅(Broadcasting)을 지원해야 합니다.

---

## 5. 대규모 트래픽 대비 백엔드 부하 분산 (Caching Strategy)

회사의 임직원 수가 늘어날 때, 통계 대시보드 및 공통 참조 데이터들이 DB 서버(PostgreSQL)에 병목(Bottleneck) 현상을 유발하는 것을 선제 방어합니다.

### 5-1. 정적 데이터 및 통계 캐싱 (백엔드 에이전트 제언)
*   **요구사항**: 전사 통계표(Analytics) 화면 호출 및 부서/카테고리(`Categories`) 목록 렌더 시, 쿼리를 매번 발생시키지 않도록 인메모리 캐싱 계층이 필요합니다.
*   **구현 스펙**: NestJS 내부에 `Redis` 기반 또는 내부 인메모리 캐시 모듈을 적용(`CacheInterceptor`)하여, 특정 데이터는 N분 내에 다시 호출될 때 DB 조회 없이 즉시 응답하도록 서버 효율성을 확보합니다.

---

### 👉 요약 및 단계별 로드맵 (Roadmap)
이상의 종합적 요구사항은 현재 MVP의 안정화를 거친 뒤 다음 3단계 우선순위로 진행될 예정입니다:
1.  **Phase 1 (보안 & 모바일)**: 파일 업로드 화이트리스트 구현 + 모바일 UI/깨짐 현상 보강.
2.  **Phase 2 (편의 & 실시간)**: 티켓 코멘트 실시간 소통 강화(WebSocket) + 인메모리 통계 캐싱.
3.  **Phase 3 (자동화)**: 업무 부하를 고려한 Auto-Routing 및 스킬 기반 할당 알고리즘 도입.
