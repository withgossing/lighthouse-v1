# Ticket Routing & Classification Review (티켓 할당 및 분류 기능 검토)

Lighthouse IT Helpdesk가 실무에서 원활하게 동작하기 위해 가장 중요한 **티켓의 업무 유형 분류(Classification)**와 **담당자 배정(Assignment 및 Auto-Routing)** 기능에 대한 분석 및 도입 제안 문서입니다.

---

## 📂 1. 업무 유형 분류 기능 (Ticket Classification)

사용자가 문제를 접수할 때, 해당 문제가 어떤 종류의 기술 지원인지를 명확히 분류하는 것은 "누가 이 문제를 가장 빠르고 정확하게 풀 수 있는가"를 결정짓는 핵심입니다.

### 1-1. 현재 스키마에 반영된 형태
현재 MVP에서는 `schema.prisma`의 `TicketCategory` Enum 값에 따라 기본적인 분류를 하고 있습니다.
*   **HARDWARE**: PC 고장, 모니터 파손, 키보드/마우스 등 물리적 장비 이슈
*   **SOFTWARE**: 오피스 프로그램 설치 에러, 사내 메신저 접속 불가 등
*   **NETWORK**: "인터넷이 안돼요", "VPN 접속 실패" 등 네트워크 인프라 이슈
*   **ACCESS**: "특정 그룹웨어 게시판 권한이 없어요", "Gitlab 계정 생성해주세요" 등 계정/권한 부여
*   **OTHER**: 분류하기 어려운 기타 장애

### 1-2. 고도화 제안 (추후 반영)
*   *문제점*: 회사가 커지면 5가지 분류만으로는 부족합니다. 또한 카테고리는 회사 정책에 따라 계속 변하므로 Enum이 아닌 별도의 `TicketCategory` 테이블로 분리하는 것이 유지보수에 유리합니다.
*   *서브 카테고리(Sub-Category)*: `대분류(하드웨어) -> 중분류(노트북) -> 소분류(배터리)` 형태로 다단계 분류(Tree)를 지원하면, 나중에 관리자(Admin)의 전문 분야별로 매칭하기가 훨씬 쉬워집니다.

## 🙋‍♂️ 2. 담당자 수동 지정 및 배분 기능 (Manual Assignment)

가져온 티켓을 특정 IT 관리자(Admin)에게 배정하는 가장 기본적인 기능입니다.

### 2-1. 구현 로직 (API & UI 측면)
*   **사용자 플로우**:
    1.  티켓이 처음 생성되면 기본적으로 담당자(`assigneeId`)가 없는 상태, 즉 **"미할당(Unassigned)"** Queue에 들어갑니다.
    2.  수석 관리자(팀장) 또는 당번 관리자가 관리자 대시보드(`/admin/tickets`)에서 미할당 티켓들을 봅니다.
    3.  티켓의 상세 화면(`/admin/tickets/[id]`) 우측 패널에서 "담당자 지정" 드롭다운을 열어, 현재 접속 중이거나 여유가 있는 다른 관리자를 선택하여 즉시 배정(Patch API) 합니다.
*   **개인 뷰(My Work)**: 관리자 본인에게 할당된 티켓만 모아서 볼 수 있는 탭("Assigned to Me")이 필요합니다.

## 🤖 3. 기존 업무 분류 및 비율 기반 자동 배정 (Auto-Routing/Round-Robin)

티켓 접수량이 많아지면 미할당 큐를 팀장이 매번 쳐다보고 수동으로 나누는 것은 큰 병목(Bottleneck)이 됩니다. 이를 해결하기 위한 "자동 배정" 로직 검토입니다.

### 3-1. Round-Robin (순차 분배) 방식
*   *개념*: 가장 단순한 방식. A, B, C 관리자가 있다면 `A -> B -> C -> A -> B` 순서로 들어오는 티켓을 순서대로 할당합니다.
*   *장점*: 구현이 매우 쉽습니다.
*   *단점*: 각 티켓의 난이도나 분류를 무시하며, 관리자 A는 5분만에 해결하고 관리자 B는 며칠 걸리는 건을 잡는 등 업무 부하(Load)의 불균형이 올 수 있습니다.

### 3-2. Workload & Skill 기반 분배 로직 (알고리즘 제안)
귀하께서 말씀하신 "기존 업무 분류 비율에 따라 배정을 해주는 기능"을 이상적으로 구현하려면 다음 로직이 필요합니다.

1.  **스킬셋(Skillset) 매핑**:
    *   DB에 각 관리자(User.Role = ADMIN)가 잘하는 분야를 매핑해둡니다. (예: `Admin A` = NETWORK 전문, `Admin B` = SOFTWARE/ACCESS 전문)
    *   새 티켓이 **NETWORK** 카테고리로 들어오면 1차적으로 `Admin A`에게 할당 후보가 좁혀집니다.
2.  **현재 부하량(Current Queue Load) 계산**:
    *   단순히 총 티켓 수가 아니라, "현재 상태가 `OPEN` 이거나 `IN_PROGRESS` 인 티켓의 수"를 관리자별로 조회(`SELECT COUNT(...) WHERE assigneeId = XX`)합니다.
    *   여기에 **우선순위 가중치(Priority Weight)**를 둬서 부하를 계산합니다. (예: `LOW` = 1점, `HIGH` = 3점, `URGENT` = 5점 등의 Point 제도를 도입.)
3.  **가드레일(Max Capacity)**:
    *   아무리 NETWORK 전문이라도, `Admin A`가 이미 10개의 티켓이 밀려있다면(Limit 초과), 차순위로 일반(OTHER) 처리를 담당하는 분에게 "우회(Fallback)"시키는 로직이 필요합니다.

### 🛠️ 당장의 MVP 적용 여부 검토
*   **MVP 1단계(현재)**: 먼저 **2번 항목("수동 할당")** 과 "미할당 큐 뷰어"를 튼튼하게 만드는 것에 집중해야 합니다. 백엔드에서 `PATCH` API를 통해 관리자 간 권한 검증을 통과하여 티켓을 주고받는 로직이 우선입니다.
*   **MVP 2단계(고도화)**: 수동 할당이 안정화되고 나면, 티켓 생성(`POST /api/tickets`) 로직 내부 혹은 별도의 백엔드 Queue Worker 시스템(예: Redis 기반 BullMQ)을 붙여 3번의 자동 평가 알고리즘을 도입하는 것이 좋습니다.
