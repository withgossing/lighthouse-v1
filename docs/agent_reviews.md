# AI 에이전트 프로젝트 검토 의견 종합 (Agent Reviews)

본 문서는 사내 IT Helpdesk 시스템(Lighthouse)의 현재 구현 상태(MVP)에 대해 5명의 사내 AI 에이전트(PM, UI/UX 디자이너, 프론트엔드, 백엔드, QA/보안)가 소스 코드와 아키텍처를 검토하고 논의한 종합 리뷰 문서입니다.

## 1. PM & 총괄 아키텍트 (PM & Architect Agent)
*   **현재 상태 리뷰**: "프론트엔드와 백엔드를 물리적으로 분리한 모노레포(Monorepo) 전략은 향후 서비스 확장을 고려했을 때 매우 훌륭한 선택입니다. 특히 `Category` 모델을 2-Depth 계층 구조로 가져가고 부서/권한별 접근 규칙(`CategoryAccess`)을 둔 것은 엔터프라이즈 사내 시스템에 꼭 필요한 유연성을 제공합니다."
*   **향후 과제 제언**: "미할당 큐(Unassigned)와 담당자 수동 배정 기능까지는 잘 구현되었습니다. 다음 고도화 단계에서는 IT 부서의 업무 부하(Load)를 줄이기 위해, 티켓 접수 시 담당자의 스킬셋과 현재 진행 중인 티켓 개수를 계산해 자동으로 배분해 주는 라운드로빈(Round-Robbin) 형태의 `Auto-Routing` 시스템 개발을 우선순위에 두어야 합니다."

## 2. UI/UX 디자이너 (UI/UX Designer Agent)
*   **현재 상태 리뷰**: "대시보드 상단 요약 카드에서 일반 사원(개인 현황만)과 관리자(전사 현황 및 미할당 알림)의 뷰포트를 이원화하여 노출 정보를 제한한 접근법이 좋습니다. 색상 배색 또한 Tailwind CSS에 내장된 색상표를 활용해 티켓 상태(초록-해결, 노랑-진행 등)를 뱃지 디자인으로 일관되게 표현하여 정보의 가독성을 높였습니다."
*   **향후 과제 제언**: "관리자가 티켓 데이터 테이블을 볼 때 컬럼이 7개(ID, 상태, 우선순위, 제목, 분류, 담당자, 생성일)에 달해 모바일(스마트폰) 환경에서는 화면 바깥으로 표가 밀리거나 깨질 가능성이 높습니다. 화면 해상도에 따라 부차적인 데이터 컬럼은 숨기는 반응형 처리(`hidden md:table-cell`)가 반드시 점검되어야 합니다."

## 3. 프론트엔드 개발자 (Frontend Developer Agent)
*   **현재 상태 리뷰**: "Next.js 15의 Server Component(RSC)를 활용해 클라이언트 측의 자바스크립트 번들 사이즈를 줄인 것은 퍼포먼스 측면에서 환영할 만합니다. `lib/data/` 내부의 fetch 함수들을 통해 데이터를 서버사이드에서 병렬 리졸브(`Promise.all`)하여 대시보드 렌더링 속도를 상당히 끌어올렸습니다."
*   **향후 과제 제언**: "상태(Status)나 우선순위(Priority)를 바꿀 때마다 페이지를 전환하지 않고 현재 페이지에서 UI가 즉시 반영되는 Optimistic UI 등의 패턴이 필요할 수 있습니다. 특히 대화형 티켓 코멘트 영역은 향후 WebSocket이나 폴링(Polling)을 붙여 사용자 경험을 메신저급으로 끌어올리는 것도 좋은 안입니다."

## 4. 백엔드 및 API 개발자 (Backend Developer Agent)
*   **현재 상태 리뷰**: "NestJS 11 버전을 채택하여 `Controller`, `Service`, `Module` 단위로 관심사를 엄격하게 분리(SoC)해 둔 아키텍처는 백엔드 팀이 협업하기 가장 이상적인 형태입니다. 특히 `TicketHistory` 엔티티를 별도로 두어, 상태 변경 및 담당자 이관 등 주요 이벤트를 모두 로깅하는 설계는 시스템 추적(Audit)에 완벽하게 대응합니다."
*   **향후 과제 제언**: "현재는 각 요청마다 DB를 직접 히트(Hit)하고 있습니다. 나중에 `GET /api/categories` 같은 정적 분류 데이터 테이블이나, 전사 대시보드 통계 계산 쪽은 Redis와 같은 인메모리 캐시를 도입하여 DB 부하를 분산시킬 준비(Cache Interceptor)를 해야 할 것입니다."

## 5. QA 및 보안 엔지니어 (QA & Security Engineer Agent)
*   **현재 상태 리뷰**: "단순히 프론트엔드 버튼 메뉴를 숨기는 것을 넘어, API 서버단에서도 전역 `@UseGuards(AuthGuard)`를 적용하고 `x-user-id`, `x-user-role` 헤더를 통해 모든 트랜잭션에서 사용자 권한을 검증하는 이중 방어 체계가 매우 단단합니다. 로컬 테스트용 모의 계정(mock-login) 세팅도 원활한 테스트 코버리지를 올리는 데 기여했습니다."
*   **향후 과제 제언**: "에러 로그 캡처 등을 위해 사용하는 파일 업로드 기능(`Attachment`)의 보안 점검이 시급합니다. 무작위 파일 업로드(웹 쉘, 악성 스크립트 등)를 막기 위해 클라이언트 및 NestJS 파이프 단에서 File Mime-type에 대한 강력한 화이트리스트 검사(예: 이미지 포맷 및 PDF만 허용)와 파일 크기 제한을 확실히 적용해야 합니다."
